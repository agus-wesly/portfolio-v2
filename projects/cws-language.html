The First Step : Scanning and Representing Code

A compiler or interpreter needs to recognize something called a token. A token is the smallest unit of meaning in a programming language that the interpreter can understand. This includes keywords like if, else, and while, as well as literals such as numbers and strings. The first step in building an interpreter is to break the source code into these tokens, a process known as tokenizing or lexing.

Once we can successfully identify each token, the next step is to transform them into a structured representation. In this case, we use an Abstract Syntax Tree (AST). Essentially, it‚Äôs a hierarchical model of the code that represents how different parts relate to each other. This structure makes it possible to both evaluate expressions and check whether the syntax is valid. The image below illustrates how an AST can be used to perform a simple arithmetic operation.

Functions

Since the language we‚Äôre building is dynamic, functions become a lot more interesting. This is because we have something called a closure, which allows a function to access variables from its outer scope even after the outer function has finished executing.

We solve this by using a chain of environments. When a function is declared, it captures the environment from the point of its declaration. Each environment also holds a reference to its parent or outer environment. Then, when the function is called, it uses the environment that was captured at the time it was defined.

The Second Version

The second version is a complete rewrite of the first one. This time, instead of using an AST, which is slower and more resource-intensive, we use bytecode along with a virtual machine. This approach is both faster and more efficient, but also more advanced. It requires a language with low-level access like C, since we need to manage memory ourselves. Virtual Machine

A virtual machine, in this context, is simply a program that reads and executes instructions in the form of bytecode. Bytecode itself is a set of instructions represented in bytes (8-bit values, as the name suggests). Instead of building an AST to represent the code, we create an array of bytes that serves as the program‚Äôs instructions. This representation is much smaller in size since it only deals with raw bytes, and it‚Äôs also faster to execute because we‚Äôre processing them directly.

Hash Map

A hashmap is one of the data structures I use most often, right after arrays. But until now, I never really understood how it worked under the hood. When working with C, we don‚Äôt get fancy data structures built in, so we have to create them ourselves. Fortunately, the book covers that topic as well, and it gave me the perfect opportunity to finally learn how a hashmap actually works.

A hashmap is basically like an array, except that it uses strings as keys instead of numbers. The problem is that indexing with strings is slow and inefficient. To make it faster, we still want to use numbers for indexing, which means we need a way to convert each string key into a numeric representation.

That‚Äôs where the hash function comes in. A hash function takes a string and produces a number based on it. It needs to be stable, meaning it should always return the same number for the same input string. Once we get that number, we use the modulo operator to determine the position where the item should be stored inside the array.

One of the problems you encounter when building a hashmap is a collision, which happens when two different values end up in the same position inside the array. There are two common ways to handle this: chaining and open addressing. Chaining works by storing multiple items in the same slot, where each item holds a reference to the next one using a linked list. This approach works well, but it requires extra memory to store pointers and introduces some pointer overhead.

Open addressing, on the other hand, doesn‚Äôt use linked lists. Instead, it stores the item in the next available slot when a collision occurs. The book uses this approach, and it comes with some nice advantages. It‚Äôs more memory-efficient and faster since we don‚Äôt have to store pointers or deal with pointer overhead. However, it does require extra care, especially when implementing features like deletion in a hashmap.
